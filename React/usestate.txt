Level 1: Basics
1. What is useState?

Answer:

useState is a React Hook that allows functional components to store and manage state.

State is data that can change over time. When the state changes, React re-renders the component to display the updated UI.

Example:

import { useState } from "react";

function App() {
  const [count, setCount] = useState(0);

  return <h1>{count}</h1>;
}

Here:

count is the current state.
setCount updates the state.
2. Why do we use useState?

Answer:

We use useState to store values that can change while the application is running.

Examples:

Counter value
User name
Form inputs
Login status
Theme (Dark/Light)
Shopping cart items
Todo list

Without useState, React wouldn't know that the data changed, so the UI wouldn't update automatically.

3. What does useState return?

Answer:

useState returns an array with two elements:

The current state value.
A function to update that state.

Example:

const [count, setCount] = useState(0);

This is equivalent to:

const stateArray = useState(0);

// stateArray[0] → current value
// stateArray[1] → update function
4. What are the two values returned by useState?

Answer:

const [count, setCount] = useState(0);
First value
count

It stores the current state.

Initially:

count = 0
Second value
setCount

It updates the state.

Example:

setCount(5);

Now React updates:

count = 5

and re-renders the component.

5. What is the syntax of useState?

Answer:

const [state, setState] = useState(initialValue);

Example:

const [count, setCount] = useState(0);

For a string:

const [name, setName] = useState("");

For a boolean:

const [isLoggedIn, setIsLoggedIn] = useState(false);

For an array:

const [numbers, setNumbers] = useState([]);

For an object:

const [user, setUser] = useState({
  name: "Vibha",
  age: 25,
});
6. Why is useState called a Hook?

Answer:

useState is called a Hook because it lets a functional component hook into React's state management.

Before Hooks, only class components could manage state.

Hooks made it possible for functional components to use state and other React features.

Example:

const [count, setCount] = useState(0);

The component is "hooking into" React's state system.

7. Where can you use useState?

Answer:

useState can only be used:

Inside a React functional component.
Inside a custom Hook.

✅ Correct:

function App() {
  const [count, setCount] = useState(0);

  return <h1>{count}</h1>;
}

Also correct:

function useCounter() {
  const [count, setCount] = useState(0);

  return { count, setCount };
}

❌ Incorrect:

const [count, setCount] = useState(0);

outside a component.

8. Can useState be used inside a class component?

Answer:

No.

useState works only with functional components.

Class components manage state differently:

class App extends React.Component {
  state = {
    count: 0,
  };
}

Functional components use:

const [count, setCount] = useState(0);
9. Can we use multiple useState Hooks in one component?

Answer:

Yes.

A component can use as many useState Hooks as needed.

Example:

function App() {
  const [name, setName] = useState("");
  const [age, setAge] = useState(25);
  const [isLoggedIn, setIsLoggedIn] = useState(false);

  return (
    <>
      <h2>{name}</h2>
      <h2>{age}</h2>
      <h2>{isLoggedIn ? "Logged In" : "Logged Out"}</h2>
    </>
  );
}

Each state variable is independent.

10. What happens if you don't use useState and use a normal variable instead?

Answer:

A normal variable does not persist between renders and changing it does not trigger a re-render.

Example:

function App() {
  let count = 0;

  const increase = () => {
    count++;
    console.log(count);
  };

  return (
    <>
      <h1>{count}</h1>
      <button onClick={increase}>Increase</button>
    </>
  );
}
What happens?

Initially:

count = 0

Click the button:

console.log → 1

But the UI still shows:

0

Why?

Because changing a normal variable doesn't tell React to re-render the component.

The correct approach is:

function App() {
  const [count, setCount] = useState(0);

  return (
    <>
      <h1>{count}</h1>
      <button onClick={() => setCount(count + 1)}>
        Increase
      </button>
    </>
  );
}

Now each click updates the state, React re-renders the component, and the UI displays the new value.

Interview tip

A common interview question is: "Why use useState instead of a normal variable?"

A concise answer is:

useState stores data between renders and tells React to re-render the component when the state changes. A normal variable is recreated on every render and changing it does not update the UI.

Level 2: State Updates
11. What happens when you call the setter function?

Answer:

When you call the setter function (like setCount), React:

Schedules a state update.
Stores the new value.
Re-renders the component.
Updates the UI with the new state.

Example:

const [count, setCount] = useState(0);

const increase = () => {
  setCount(count + 1);
};

Initially:

count = 0

Click button:

setCount(1)

React updates state and re-renders.

New UI:

1
12. Does changing state re-render the component?

Answer:

✅ Yes.

Whenever state changes, React re-renders the component so the UI reflects the latest state.

Example:

const [name, setName] = useState("Vibha");

setName("Rahul");

React renders again and displays:

Rahul
13. Does updating a normal variable re-render the component?

Answer:

❌ No.

Example:

let count = 0;

const increase = () => {
  count++;
};

Even though count changes, React doesn't know about it, so the UI does not update.

Only state updates trigger re-renders.

14. Is state updated immediately?

Answer:

❌ No.

State updates are scheduled by React and applied before the next render.

Example:

const [count, setCount] = useState(0);

const handleClick = () => {
  setCount(1);

  console.log(count);
};

Output:

0

because count hasn't updated yet during the current function execution.

After React re-renders:

count = 1
15. Why is a state update considered asynchronous?

Answer:

React schedules state updates instead of applying them instantly. This allows React to:

Batch multiple updates together.
Avoid unnecessary re-renders.
Improve performance.

So after calling:

setCount(10);

React doesn't immediately change count. It updates it during the next render cycle.

16. Can you read the updated state immediately after calling setState?

Answer:

❌ No.

Example:

const [count, setCount] = useState(0);

const handleClick = () => {
  setCount(5);

  console.log(count);
};

Output:

0

because React hasn't re-rendered yet.

On the next render:

count = 5
17. What happens if you call setState multiple times?

Answer:

React processes all state updates.

Example:

setCount(1);
setCount(2);
setCount(3);

Final state:

3

The last update wins because each call replaces the previous value.

However, if the next state depends on the previous one, use the functional updater:

setCount(prev => prev + 1);
setCount(prev => prev + 1);
setCount(prev => prev + 1);

Now each update builds on the previous one.

18. What is React batching?

Answer:

Batching means React groups multiple state updates into a single re-render.

Example:

const handleClick = () => {
  setName("Vibha");
  setAge(25);
  setCity("Pune");
};

Instead of rendering three times, React batches these updates and renders once.

Benefits:

Better performance.
Fewer unnecessary renders.
Smoother UI updates.
19. Why is batching useful?

Answer:

Without batching:

setName()

↓

Render

↓

setAge()

↓

Render

↓

setCity()

↓

Render

Total:

3 renders

With batching:

setName()
setAge()
setCity()

↓

1 render

Advantages:

Faster applications.
Better performance.
Less work for React.
20. Does React batch updates only in event handlers?

Answer:

❌ No.

Starting with React 18, React automatically batches updates in many situations, including:

Event handlers
setTimeout
Promises
Async functions
Native event handlers

Example:

setTimeout(() => {
  setCount(c => c + 1);
  setName("Vibha");
}, 1000);

In React 18, these updates are batched into a single render.

Level 3: Functional Updates
21. What's the difference between these?
setCount(count + 1);

and

setCount((prev) => prev + 1);
Answer

Both increment the count, but they behave differently when multiple updates happen.

1. Direct update
setCount(count + 1);

Uses the current value of count from the current render.

Example:

const [count, setCount] = useState(0);

const handleClick = () => {
  setCount(count + 1);
};

If count = 0

After click:

count = 1
2. Functional update
setCount((prev) => prev + 1);

Uses the latest state value provided by React.

Example:

const handleClick = () => {
  setCount((prev) => prev + 1);
};

If count = 0

After click:

count = 1

Both give the same result for one update, but the functional form is safer when the next state depends on the previous state.

22. When should you use the functional updater?
Answer

Use it whenever the new state depends on the previous state.

Examples:

Increment counter
Decrement counter
Toggle boolean
Add item to an array
Remove item from an array
Update an object based on its current values

Example:

setCount((prev) => prev + 1);
setIsOpen((prev) => !prev);
setItems((prev) => [...prev, newItem]);
23. Why is the functional updater safer?
Answer

Because React may batch multiple updates together.

Instead of using an outdated value, React passes the latest state into the updater function.

Example:

setCount((prev) => prev + 1);

Even if several updates happen, prev is always the most recent value.

24. What does prev represent?
Answer

prev is the previous (current/latest) state value that React provides.

Example:

const [count, setCount] = useState(5);

setCount((prev) => prev + 1);

Here:

prev = 5

React returns:

6

After re-render:

count = 6
25. What happens here?
setCount(count + 1);
setCount(count + 1);

Assume:

count = 0;

Both lines evaluate to:

setCount(1);
setCount(1);

React processes them, and the final state is:

count = 1

Not

count = 2

because both updates use the same old value (0).

26. What happens here?
setCount((prev) => prev + 1);
setCount((prev) => prev + 1);

Assume:

count = 0

First update:

0 → 1

Second update:

1 → 2

Final state:

count = 2

Each updater receives the latest state.

27. Can you rename prev?

Example:

setCount((current) => current + 1);
Answer

✅ Yes.

prev is just a parameter name.

These are all valid:

setCount((prev) => prev + 1);
setCount((current) => current + 1);
setCount((value) => value + 1);
setCount((oldCount) => oldCount + 1);

Most developers use prev because it's short and descriptive.

28. Is prev a keyword?
Answer

❌ No.

It's a regular function parameter.

Example:

setCount((banana) => banana + 1);

This also works because banana is just the parameter name.

29. Why is the functional updater preferred in production?
Answer

Because it avoids bugs caused by stale state and batching.

Example:

setLikes((prev) => prev + 1);

This always uses the latest value, even when multiple updates are queued.

Benefits:

More reliable
Works correctly with batched updates
Easier to maintain
Recommended by the React team when the new state depends on the old state
30. Which approach is recommended in interviews?
Answer

It depends on the situation.

If you're replacing the state with a new value:

setName("Vibha");

or

setIsLoggedIn(true);

this is perfectly fine.

If you're calculating the next state from the previous one:

setCount((prev) => prev + 1);
setIsOpen((prev) => !prev);
setItems((prev) => [...prev, newItem]);

use the functional updater.

Interview rule to remember

If the new state depends on the previous state, use the functional updater. Otherwise, pass the new value directly.

36. Can state store functions?
Answer

✅ Yes, but with an important distinction.

You can store a function in state, but if you pass a function directly to useState, React treats it as a lazy initializer.

To store a function:

const [multiply] = useState(() => {
  return (x) => x * 2;
});

console.log(multiply(5)); // 10

If you do this:

const [value] = useState(() => 100);

React calls the function once and stores 100, not the function itself.

Interview Tip: Most applications don't store functions in state. They usually store data.

38. Can state store undefined?
Answer

✅ Yes, but it's generally not recommended.

Example:

const [value, setValue] = useState(undefined);

or simply:

const [value, setValue] = useState();

However, it's usually better to use a more meaningful initial value:

useState("");

or

useState(null);

or

useState([]);

This makes your code easier to understand and avoids unnecessary checks.

39. Can state store nested objects?
Answer

✅ Yes.

Example:

const [user, setUser] = useState({
  name: "Vibha",
  address: {
    city: "Pune",
    pincode: 411001,
  },
});

Update only the city:

setUser((prev) => ({
  ...prev,
  address: {
    ...prev.address,
    city: "Mumbai",
  },
}));

Notice that you spread both the outer object and the nested object to avoid mutating state.

40. Can state store arrays of objects?
Answer

✅ Yes.

This is one of the most common patterns in React.

Example:

const [students, setStudents] = useState([
  { id: 1, name: "Amit" },
  { id: 2, name: "Vibha" },

setStudents((prev) => [
  ...prev,
  { id: 3, name: "Rahul" },

setStudents((prev) =>
  prev.map((student) =>
    student.id === 2
      ? { ...student, name: "Priya" }
      : student
  )
);

Delete a student:

setStudents((prev) =>
  prev.filter((student) => student.id !== 1)
);

Use cases:

Employee lists
Product lists
Todo apps
Shopping carts
Chat messages

Level 5: Arrays
41. Why shouldn't you use push() directly on state?
Answer

push() mutates (changes) the original array.

React expects state to be treated as immutable, meaning you should create a new array instead of changing the existing one.

❌ Wrong:

const [numbers, setNumbers] = useState([10, 20, 30]);

numbers.push(40);
setNumbers(numbers);

✅ Correct:

setNumbers((prev) => [...prev, 40]);

The spread operator (...) creates a new array, allowing React to detect the change.

42. Why is this wrong?
numbers.push(5);
setNumbers(numbers);
Answer

push() changes the original array.

Example:

Before:

[10, 20, 30]

After:

numbers.push(5);

The same array becomes:

[10, 20, 30, 5]

Since the reference hasn't changed, React may not recognize it as a new state, which can lead to bugs or skipped re-renders.

Instead, create a new array:

setNumbers((prev) => [...prev, 5]);
43. Correct way to add an item?
Answer

Use the spread operator.

setNumbers((prev) => [...prev, 40]);

Example:

Before:

[10, 20, 30]

After:

[10, 20, 30, 40]

You can also add at the beginning:

setNumbers((prev) => [5, ...prev]);

Result:

[5, 10, 20, 30]
44. Correct way to remove an item?
Answer

Use filter().

Example:

const [numbers, setNumbers] = useState([10, 20, 30, 40]);

setNumbers((prev) =>
  prev.filter((num) => num !== 20)
);

Result:

[10, 30, 40]

Or remove by index:

setNumbers((prev) =>
  prev.filter((_, index) => index !== 1)
);
45. Correct way to update an item?
Answer

Use map().

Example:

const [numbers, setNumbers] = useState([10, 20, 30]);

setNumbers((prev) =>
  prev.map((num) =>
    num === 20 ? 99 : num
  )
);

Result:

[10, 99, 30]

map() creates a new array while updating only the matching element.

46. Why do we use the spread operator?
Answer

The spread operator (...) copies an array or object.

Example:

const arr = [10, 20, 30];

const newArr = [...arr];

Result:

newArr = [10, 20, 30]

Now you can safely add or modify items without changing the original array.

Example:

setNumbers((prev) => [...prev, 40]);

This creates a new array:

[10, 20, 30, 40]

instead of modifying the existing one.

47. What happens if you mutate an array directly?
Answer

Direct mutation can cause:

Bugs
Unpredictable behavior
React missing the update
Difficult debugging

Example:

numbers.push(40);

or

numbers[0] = 100;

These modify the original array.

React relies on detecting a new reference to know that state has changed.

Always create a new array:

setNumbers((prev) => [...prev, 40]);
48. Which methods are commonly used with array state?
Answer
map()

Update items.

setNumbers((prev) =>
  prev.map((num) =>
    num * 2
  )
);
filter()

Remove items.

setNumbers((prev) =>
  prev.filter((num) => num !== 20)
);
slice()

Copy part of an array.

const firstTwo = numbers.slice(0, 2);

Or remove the last item:

setNumbers((prev) =>
  prev.slice(0, -1)
);
concat()

Add items without mutating.

setNumbers((prev) =>
  prev.concat(40)
);

Result:

[10, 20, 30, 40]

Other useful methods:

find()
findIndex()
some()
every()
reduce()
sort() (create a copy first: [...prev].sort())
49. Why is immutability important?
Answer

Immutability means never modifying the original state directly.

Instead of:

numbers.push(40);

Create a new array:

[...numbers, 40]

Benefits:

React can detect changes.
Components re-render correctly.
Easier debugging.
More predictable code.
Better performance through efficient change detection.
50. How do you clear an array state?
Answer

Set it to an empty array.

setNumbers([]);

Example:

Before:

[10, 20, 30]

After:

[]

Example button:

<button onClick={() => setNumbers([])}>
  Clear Array
</button>

Interview Tip

A common follow-up question is:

"Can I use sort() or reverse() on state?"

The answer is not directly, because both methods mutate the original array.

❌ Avoid:

numbers.sort();
numbers.reverse();

✅ Instead, make a copy first:

setNumbers((prev) => [...prev].sort());

setNumbers((prev) => [...prev].reverse());

This keeps your state immutable and ensures React sees a new array reference.

Level 6: Objects
51. Why shouldn't you modify an object directly?
Answer

React state should be treated as immutable.

If you modify the object directly, React may not detect the change because the object reference stays the same.

❌ Wrong:

const [user, setUser] = useState({
  name: "Vibha",
  age: 24,
});

user.age = 25;
setUser(user);

✅ Correct:

setUser((prev) => ({
  ...prev,
  age: 25,
}));
52. What's wrong with this?
user.age = 25;
setUser(user);
Answer

user.age = 25 mutates the existing object.

Before:

{
  name: "Vibha",
  age: 24
}

After:

{
  name: "Vibha",
  age: 25
}

The object reference is still the same.

React compares object references, not individual properties, so it may not recognize this as a new state.

Always create a new object instead.

53. Correct way to update an object?
Answer

Use the spread operator (...).

const [user, setUser] = useState({
  name: "Vibha",
  age: 24,
});

Update age:

setUser((prev) => ({
  ...prev,
  age: 25,
}));

Result:

{
  name: "Vibha",
  age: 25
}

The original object remains unchanged.

54. Why do we use:
{
  ...user
}
Answer

The spread operator copies all properties of the object into a new object.

Example:

const user = {
  name: "Vibha",
  age: 24,
};
const newUser = {
  ...user,
};

Result:

{
  name: "Vibha",
  age: 24,
}

Now you can safely update a property:

const updatedUser = {
  ...user,
  age: 25,
};

Result:

{
  name: "Vibha",
  age: 25,
}
55. How do you update one property?
Answer

Suppose:

const [user, setUser] = useState({
  name: "Vibha",
  age: 24,
  city: "Pune",
});

Update only the city:

setUser((prev) => ({
  ...prev,
  city: "Mumbai",
}));

Result:

{
  name: "Vibha",
  age: 24,
  city: "Mumbai",
}

Only city changes. Other properties stay the same.

56. How do you update nested objects?
Answer

Suppose:

const [user, setUser] = useState({
  name: "Vibha",
  address: {
    city: "Pune",
    pincode: 411001,
  },
});

Update only the city:

setUser((prev) => ({
  ...prev,
  address: {
    ...prev.address,
    city: "Mumbai",
  },
}));

Result:

{
  name: "Vibha",
  address: {
    city: "Mumbai",
    pincode: 411001,
  },
}

Notice that we spread both the outer object and the nested address object.

57. How do you remove a property?
Answer

Use object destructuring with the rest operator.

Suppose:

const user = {
  name: "Vibha",
  age: 24,
  city: "Pune",
};

Remove age:

setUser((prev) => {
  const { age, ...rest } = prev;
  return rest;
});

Result:

{
  name: "Vibha",
  city: "Pune",
}

Another option is to create a new object manually:

setUser({
  name: user.name,
  city: user.city,
});
58. Can object state contain arrays?
Answer

✅ Yes.

Example:

const [user, setUser] = useState({
  name: "Vibha",
  skills: ["React", "JavaScript"],
});

Add a new skill:

setUser((prev) => ({
  ...prev,
  skills: [...prev.skills, "Node.js"],
}));

Result:

{
  name: "Vibha",
  skills: [
    "React",
    "JavaScript",
    "Node.js"
  ]
}
59. Can object state contain another object?
Answer

✅ Yes.

Example:

const [user, setUser] = useState({
  name: "Vibha",
  address: {
    city: "Pune",
    state: "Maharashtra",
  },
});

This is very common in React applications.

Update only the state:

setUser((prev) => ({
  ...prev,
  address: {
    ...prev.address,
    state: "Goa",
  },
}));
60. What happens if you forget the spread operator?
Answer

Suppose:

const [user, setUser] = useState({
  name: "Vibha",
  age: 24,
});

You write:

setUser({
  age: 25,
});

Result:

{
  age: 25
}

The name property is lost because you're replacing the entire object with a new one that only has age.

The correct way is:

setUser((prev) => ({
  ...prev,
  age: 25,
}));

Result:

{
  name: "Vibha",
  age: 25,
}

The spread operator preserves existing properties while updating only the ones you specify.


Interview Tip

A very common question is:

Why does React recommend treating state as immutable?

A strong answer is:

React detects state changes by comparing references. Creating a new object or array gives React a new reference, making updates predictable and allowing React to efficiently determine when to re-render components.

Level 7: Rendering
61. What triggers a re-render?
Answer

A component re-renders when React detects that something affecting its output has changed.

Common triggers are:

State changes (useState)
Props change
Context value changes
Parent component re-renders (unless optimized with React.memo)

Example:

const [count, setCount] = useState(0);

<button onClick={() => setCount(count + 1)}>
  Increase
</button>

Clicking the button updates the state, so React re-renders the component.

62. Does every state update cause a re-render?
Answer

Usually yes, but React may skip a render if the new state is the same as the current state.

Example:

const [count, setCount] = useState(5);

setCount(5);

Since the value hasn't changed, React can skip re-rendering.

63. What happens if the new state equals the old state?
Answer

React compares the old state and new state using Object.is().

If they're the same, React skips re-rendering that state update.

Example:

const [name, setName] = useState("Vibha");

setName("Vibha");

React sees:

Old = "Vibha"
New = "Vibha"

No UI update is needed.

64. Why doesn't React always re-render?
Answer

React tries to avoid unnecessary work.

If nothing has changed, rendering again would waste CPU time and reduce performance.

Example:

setCount(10);
setCount(10);

The second update doesn't change the state, so React can ignore it.

65. How does React compare old and new state?
Answer

React uses Object.is() to compare the previous state and the new state.

Examples:

Object.is(10, 10)

Result:

true
Object.is("React", "React")

Result:

true

For objects:

Object.is(
  { name: "A" },
  { name: "A" }
)

Result:

false

because they are different object references, even though their contents are the same.

66. Does React compare object contents or references?
Answer

React compares references, not the contents.

Example:

const obj1 = { name: "Vibha" };
const obj2 = { name: "Vibha" };

console.log(obj1 === obj2);

Output:

false

Although the properties match, they are different objects in memory.

This is why React recommends creating new objects instead of mutating existing ones.

67. Why is immutability useful for rendering?
Answer

Immutability creates a new object or array reference.

React can quickly detect the new reference and know that something changed.

Example:

setUser((prev) => ({
  ...prev,
  age: 25,
}));

Benefits:

Predictable updates
Easier debugging
Better performance
Reliable re-renders
68. What happens if state is updated during rendering?
Answer

Updating state during rendering causes React to render again immediately.

If it keeps happening on every render, it creates an infinite render loop.

❌ Wrong:

function App() {
  const [count, setCount] = useState(0);

  setCount(count + 1);

  return <h1>{count}</h1>;
}

This causes:

Render
↓

setCount()

↓

Render

↓

setCount()

↓

Render

...

React eventually throws an error.

69. Can calling setState inside render cause problems?
Answer

✅ Yes.

Calling a setter directly in the component body causes continuous re-renders.

Example:

function App() {
  const [count, setCount] = useState(0);

  setCount(1);

  return <h1>{count}</h1>;
}

React will display an error such as:

Too many re-renders.
React limits the number of renders to prevent an infinite loop.

Instead, update state in:

Event handlers
useEffect
Async callbacks
API responses
70. What is an infinite render loop?
Answer

An infinite render loop happens when each render triggers another render with no stopping condition.

Example:

function App() {
  const [count, setCount] = useState(0);

  setCount(count + 1);

  return <h1>{count}</h1>;
}

Flow:

Render
↓

setCount()

↓

Render

↓

setCount()

↓

Render

↓

...

React stops it and throws:

Too many re-renders.
Correct approach

Update state in response to an event:

function App() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Increase
    </button>
  );
}

or in an effect when appropriate:

useEffect(() => {
  setCount(1);
}, []);

The empty dependency array ([]) ensures the effect runs only once after the initial render.

Answer:

✅ onClick={() => setCount(count + 1)} passes a function to onClick. The state updates only when the button is clicked.
❌ onClick={setCount(count + 1)} calls setCount immediately during rendering, which updates state while rendering and leads to repeated renders (or at least an unintended immediate update).

Level 8: Forms
71. What is a controlled component?
Answer

A controlled component is a form element whose value is controlled by React state.

Instead of the browser managing the input value, React manages it using useState.

Example:

import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  return (
    <input
      type="text"
      value={name}
      onChange={(e) => setName(e.target.value)}
    />
  );
}

Here:

value={name} → React controls the input.
onChange → Updates the state whenever the user types.
72. Why use value with onChange?
Answer

They work together to make the input a controlled component.

<input
  value={name}
  onChange={(e) => setName(e.target.value)}
/>
value displays the current state.
onChange updates the state.

Flow:

User types
      ↓
onChange runs
      ↓
setName()
      ↓
State updates
      ↓
Component re-renders
      ↓
Input displays new value

Without onChange, the input becomes read-only.

73. Why is this incorrect?
<input value={name} />

without onChange.

Answer

The input becomes read-only.

Example:

const [name] = useState("Vibha");

<input value={name} />

You cannot type because React controls the value but never updates it.

React even shows a warning similar to:

You provided a `value` prop to a form field without an `onChange` handler.

Correct:

<input
  value={name}
  onChange={(e) => setName(e.target.value)}
/>
74. How do you handle multiple input fields?
Answer

A common approach is to store all fields in a single object.

Example:

const [form, setForm] = useState({
  name: "",
  email: "",
  phone: "",
});

Update a field:

<input
  name="name"
  value={form.name}
  onChange={(e) =>
    setForm((prev) => ({
      ...prev,
      [e.target.name]: e.target.value,
    }))
  }
/>

This same handler works for all inputs as long as each input has the correct name attribute.

75. Should each input have separate state?
Answer

It depends on the form.

For a small form:

const [name, setName] = useState("");
const [age, setAge] = useState("");

This is perfectly fine.

For a larger form:

const [form, setForm] = useState({
  name: "",
  email: "",
  phone: "",
  password: "",
});

Using a single object is usually cleaner and easier to maintain.

76. When is using a single object state better?
Answer

When multiple values belong together.

Examples:

Registration form
Login form
Employee form
Profile form

Example:

const [form, setForm] = useState({
  firstName: "",
  lastName: "",
  email: "",
  password: "",
});

Benefits:

Easier to pass around as one object.
One reset clears the entire form.
Fewer state variables.
77. How do you reset a form?
Answer

Set the state back to its initial values.

Example:

const [form, setForm] = useState({
  name: "",
  email: "",
});

Reset:

setForm({
  name: "",
  email: "",
});

Button:

<button
  onClick={() =>
    setForm({
      name: "",
      email: "",
    })
  }
>
  Reset
</button>
78. How do you update only one field?
Answer

Use the spread operator.

Example:

const [form, setForm] = useState({
  name: "",
  email: "",
});

Update only email:

setForm((prev) => ({
  ...prev,
  email: "vibha@gmail.com",
}));

Result:

Before:

{
  name: "Vibha",
  email: ""
}

After:

{
  name: "Vibha",
  email: "vibha@gmail.com"
}

Only the email field changes.

79. Why do forms re-render on every keystroke?
Answer

Because every keystroke updates the state.

Example:

<input
  value={name}
  onChange={(e) => setName(e.target.value)}
/>

Typing:

H

calls:

setName("H");

Typing:

He

calls:

setName("He");

Each setName() updates state, so React re-renders the component to keep the UI in sync.

80. Is that expected?
Answer

✅ Yes.

This is normal behavior in React.

React is optimized to handle frequent re-renders efficiently.

If performance becomes an issue (for example, a very large form or expensive calculations), you can optimize using:

React.memo
useMemo
useCallback
Debouncing user input
Splitting large forms into smaller components

For normal forms, re-rendering on each keystroke is expected and not a problem.

Level 9: Tricky Questions
81. Why is this wrong?
const [user] = useState({ name: "Vibha" });

user.name = "John";
Answer

This directly mutates the state object.

Although the object's property changes, React doesn't know about it because the setter function (setUser) was never called.

❌ Wrong:

user.name = "John";

✅ Correct:

const [user, setUser] = useState({
  name: "Vibha",
});

setUser((prev) => ({
  ...prev,
  name: "John",
}));
Interview Tip

Never modify state directly. Always use the setter function.

82. What happens here?
const [count, setCount] = useState(0);

console.log(count);

setCount(1);

console.log(count);
Answer

Output:

0
0
Why?

When setCount(1) is called, React schedules the update.

The current function continues executing with the old value.

Only after React re-renders will count become:

1
Common Interview Question

Will the second console.log() print 1?

❌ No.

It still prints the old value because the component hasn't re-rendered yet.

83. Why is this incorrect?
const [count, setCount] = useState(0);

setCount(count + 1);

setCount(count + 1);

setCount(count + 1);
Answer

Assume:

count = 0

Each line becomes:

setCount(1);
setCount(1);
setCount(1);

Final result:

count = 1

Not:

3

because all three updates use the same stale value (0).

Correct way
setCount((prev) => prev + 1);
setCount((prev) => prev + 1);
setCount((prev) => prev + 1);

Now:

0 → 1 → 2 → 3

Final value:

count = 3
84. Why does this work?
setCount((prev) => prev + 1);

three times.

Answer

Each updater receives the latest state value.

Starting with:

count = 0

Flow:

First:
0 → 1

Second:
1 → 2

Third:
2 → 3

Final value:

count = 3

React executes each updater in sequence using the latest available state.

85. Can state updates be skipped?
Answer

✅ Yes.

React skips a state update if the new value is equal to the current value.

Example:

const [count, setCount] = useState(5);

setCount(5);

Since the value didn't change, React can skip the re-render.

React determines this using Object.is().

86. Why shouldn't Hooks be called conditionally?
Answer

Hooks must always be called in the same order on every render.

❌ Wrong:

if (isLoggedIn) {
  const [name, setName] = useState("");
}

If isLoggedIn changes, the order of Hooks changes.

React won't know which state belongs to which Hook.

Correct
const [name, setName] = useState("");

if (isLoggedIn) {
  console.log(name);
}

The Hook is always called; only the logic is conditional.

87. Can you call useState inside a loop?
Answer

❌ No.

Example:

for (let i = 0; i < 5; i++) {
  const [count, setCount] = useState(0);
}

This violates the Rules of Hooks because the number of Hook calls can vary.

Hooks should only be called at the top level of a component or a custom Hook.

88. Can you call useState inside an if statement?
Answer

❌ No.

Example:

if (show) {
  const [count, setCount] = useState(0);
}

If show changes between renders, the Hook order changes, causing bugs.

Correct:

const [count, setCount] = useState(0);

if (show) {
  console.log(count);
}

The Hook call stays at the top level.

89. Why must Hooks always be called in the same order?
Answer

React identifies Hooks by their call order, not by their variable names.

Example:

const [name, setName] = useState("");
const [age, setAge] = useState(20);

React remembers:

Hook #1 → name
Hook #2 → age

If you skip the first Hook on a later render:

if (show) {
  const [name, setName] = useState("");
}

const [age, setAge] = useState(20);

React now sees:

Hook #1 → age

It mistakenly assigns the previous state of name to age, leading to incorrect behavior.

90. What happens if the order changes?
Answer

React can no longer match state values with the correct Hooks.

This can cause:

Incorrect state values
Runtime errors
Unpredictable behavior

React may show an error like:

Rendered fewer hooks than expected.

or

Rendered more hooks than during the previous render.
Correct Rule

Always call Hooks:

✅ At the top level of a React function component.
✅ At the top level of a custom Hook.

Never call them:

❌ Inside if statements
❌ Inside loops
❌ Inside nested functions
❌ Inside event handlers

⭐ Bonus Interview Question

Q: Why doesn't React identify Hooks by variable names?

Example:

const [count, setCount] = useState(0);
const [name, setName] = useState("");

React does not store:

count
name

Instead, it internally tracks them by position:

Hook #1 → useState(0)
Hook #2 → useState("")

That's why renaming variables is fine:

const [value, setValue] = useState(0);

works exactly the same as:

const [count, setCount] = useState(0);

The variable names are for your code only; React relies on the order in which Hooks are called.
